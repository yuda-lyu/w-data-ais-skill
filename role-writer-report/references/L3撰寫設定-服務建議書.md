# L3 撰寫設定 — 服務建議書

本檔為「服務建議書」之**具體執行細則**（L3），於 L1（角色）+ L2（原則）之上，補充服務建議書撰寫時之**具體 Do/Don't 清單**與**操作細則**。

**檔案層級**：
- L1 L1角色設定.md：跨報告類型通用之角色與風格基底
- L2 L2報告設定-服務建議書.md：服務建議書之原則性設定
- **L3（本檔）**：服務建議書之具體 Do/Don't 清單與操作細則

**讀檔順序**：L1 → L2 → L3。L1 提供「人」與「風格」，L2 提供「原則」，L3 提供「執行細則」。L3 之細則皆為 L1/L2 原則之具體落地。

## 一、章節結構

### 1.1 三層骨架（每一工項必備）

每一工項採三段式：

> **(1) 研究背景與目的**、**(2) 工作內容與方法**、**(3) 工作預期成果**、**(4) 參考文獻**

服務實施計劃書另加「年度預期成果」（具體年份區段依當期契約填入）。本 L3 為服務建議書，不含此項。

#### 「研究背景與目的」段之構成要素

採兩段式論述：

1. **第一段（脈絡與目標）**：陳述本工作之計畫脈絡（多年期、多分項、多團隊）、所產出之資料種類、所要達成之核心特性／要素（如格式之一致性、可關聯性、可追溯性等三大核心要素），以及對主計畫之意義與後續應用之效益。此段以**正面論述為主**。
2. **第二段（既有基礎與本工項立意）**：說明承接前期已建之基礎、本工項擬於其上之擴充方向、以及兼顧之多重需求（內部資料治理、研究支援、對外溝通等）。

**「不做之風險」段為選擇性**：

- 寫入條件（須同時滿足）：
  1. 風險屬「業主端在無此工項時之本質困難」（如「跨年度交接斷鏈」、「驗收依據不足」、「需求認知不一」、「設計與實作脫節」）
  2. 風險具體、明確、且具威脅性，非抽象推論
- 禁寫類型：
  - 「乙方未建構 X 流程／機制 → 業主端 Y 風險」之 likelihood 句（違反 L2 §三角色邊界）
  - 抽象推論性風險（「資料失真」、「對外溝通誤解」、「揭露邊界鬆動」、「信賴度損害」）
  - 過度恫嚇之風險（「整體系統面臨崩潰」、「不可挽回之損失」）
- 若不滿足寫入條件，**整段省略**，背景段以正面論述為主。

**長度建議**：每節 (1) 全文 250–400 字；超過 600 字者通常為防禦填料過多，須精簡。

### 1.2 章節編號處理

#### 標題編號

- **標題沿用分寫指引之相對編號**（如 `# 1` / `## 1.1` / `### 1.1.1` / `#### (1)`），使用 markdown `#` 標頭。
- 主報告彙整時，主編彙整人員會將相對編號替換為主報告中之絕對編號（如 `3.2.7`），此為主編職責，AI 撰寫階段**不需處理絕對編號**。

#### 內文不得出現章節編號

- 內文若需列舉多節名稱或提及多項工作，**串成一句話陳述，不要分列**；各節名稱以中文引號 `「」` 包住。

**錯誤範例**（分列且寫死編號）：

```markdown
本工作範圍依招標需求分為下列四大節：
- 1.1 資料庫系統功能擴充與管理
- 1.2 對外展示網站建置
- 1.3 系統資安功能維護與更新
- 1.4 系統測試、使用與運維文件
```

**正確範例**（串成一句、引號包住、無編號）：

```markdown
本工作範圍依招標需求分為「資料庫系統功能擴充與管理」、「對外展示網站建置」、「系統資安功能維護與更新」與「系統測試、使用與運維文件」等四大項。
```

#### 交叉引用他節時以節名描述

不寫「詳見 1.3 節」；改寫「詳見本工作『系統測試與運維』章節」。但**注意**：此類跨章節引用本身應節制使用（見 L1 §5.2 不做跨章節充文字式引用）。

### 1.3 節內 (1)(2)(3)(4) 層級一致性

- 各節內所含之 `(1) 研究背景與目的`、`(2) 工作內容與方法`、`(3) 工作預期成果`、`(4) 參考文獻` 四小項之 markdown 標題層級，應於**全文跨節保持一致**。
- 判斷方式：以本分寫範圍內含最深層子節者為基準。典型為含有第三層子節（如 `1.1.1`）之節，其 (1)(2)(3)(4) 使用 `#### (1)` 層級。
- 未含第三層子節之節（如 `1.2` 直接下接 (1)(2)(3)(4)），其 (1)(2)(3)(4) **仍應使用 `#### (1)` 層級**（與其他節齊平），不可退化為 `### (1)`。

## 二、(3) 工作預期成果與 (2) 工作內容方法之顆粒度對應

### 2.1 規則

(3) 工作預期成果之條目數與順序，**應與 (2) 工作內容與方法之頂層條目一一對應**：

- (2) 有 (a)–(h) 八項，(3) 也應有 (a)–(h) 八項，各自呼應同一序號之 (2) 項目。
- 不得將 (2) 多條彙整為 (3) 單條成果。
- **例外**：若某條 (2) 工作不適合單獨列為成果（如「分階推進與年度檢討」之治理性／管理性工作），於 (3) 末尾合併為一條總結，或於 (3) 完全省略。

### 2.2 反例與正例

**反例**（(2) 5 主題模組 + 3 項其他工作 = 8 條，(3) 僅 5 條彙整）：

```markdown
(2) 工作內容與方法
(a)-(e) 5 個主題模組...
(f) 查詢效能優化
(g) UI/UX 優化
(h) 數據管理擴充

(3) 工作預期成果
(a) 完成 5 主題模組之擷取整合 ← 將 (2) 之 5 條彙整為 1 條，違規
```

**正例**（(2) 與 (3) 之主項各自 1:1）：

```markdown
(3) 工作預期成果
(a) 針對指定國內外資料格式進行匯入、支援與整合
(b) 針對指定主題 B 之資料進行客製化擷取與呈現
(c) 針對指定主題 C 之統計與模擬參數進行客製化提取與視覺化呈現
(d) 針對指定主題 D 之規格參數進行客製化擷取與呈現
(e) 針對指定主題 E 之分析參數整理與資訊呈現
(f) 完成改善資料庫之資料查詢效率與優化使用者界面功能...
(g) 妥善基於既有技術架構，持續優化資料庫之數據管理機制與功能
```

## 三、Do 清單（撰寫時應主動採用）

### 3.1 範疇之回應

- **D1**：對招標明列之每一項目逐項落地、逐項回應；不漏項。
- **D2**：以「本工作將基於既有……持續推進與深化」、「於既有基礎上擴充」、「延續既有技術線與架構」等句型錨定承諾範圍於既有可達之上。
- **D3**：具體成品、清單、頻次、規格以「依會議結論辦理」、「依規定辦理」、「依招標需求辦理」、「適時提出討論」等緩衝句承載。

### 3.2 動詞之分配（呼應 L1 §2.5）

- **D4**：招標明定必做之動作用「**將**」（例：「將依招標需求每季更新…」、「將提交…報告」）。
- **D5**：自選之實作方式用「**擬**」（例：「擬採響應式網頁設計」、「擬規劃以…為原則」）。
- **D6**：「將」與「擬」比例約 **2.5 : 1**，不超過 3 : 1。
- **D7**：同段落多句承諾，**交錯使用**「將／擬／提交／辦理／建置／規劃／配合」等多元動詞為句首動作；句首「本工作將…」連續禁 ≥ 3 次。

### 3.3 連接詞之偏好（呼應 L1 §2.3）

- **D8**：最優先積極使用「**亦**」、「**除…亦…**」之段內補述句型；密度約每千字 1.7 次。
- **D9**：「**並**」、「**並基於**」、「**並配合**」用於並列動作。
- **D10**：「**另**」段落起首每節限一次以內。

### 3.4 緩衝句之多元化（避免單一化）

- **D11**：第一順位短句須**交錯使用**，避免單一句型於同章節重複 ≥ 3 次：
  - 「依會議結論辦理」
  - 「依規定辦理」
  - 「依招標需求辦理」
  - 「適時提出討論」、「適時檢討調整」
  - 「依當期契約」、「依招標條文」
  - 「依業主之指示辦理」（節制使用，全文不超過 3 次）
  - 「當期主流」、「現行版本」
  - 「包含但不限於」
- **D12**：避免「據以辦理／據以定案／據以實施」於同段落連用 ≥ 3 次；改用第一順位短句交錯。

### 3.5 會議用語

- **D13**：本案計畫期間之會議分為定期會議與非定期會議兩類，總稱「**定期/非定期會議**」（為本案優先用語）；全文至少出現 2-3 次。
- **D14**：「需求討論會議」、「指定會議」為等價變式並用，無強制統一。

### 3.6 主語之偏好（呼應 L1 §2.4）

- **D15**：首選「**本工作**」；少量「**本工項**」於指涉葉節時。
- **D16**：避免「本計畫」；禁用「本章」、「本節」。

### 3.7 共通技術機制之描述

- **D17**：身分驗證、版次紀錄、加密／備援等共通技術機制於相關章節**各自落地一次**（單句）即可，不於開頭段或多節落點集中列舉。

### 3.8 圖示處理

- **D18**：AI 產製階段**不主動寫「如圖 X 所示」之引用、不插入圖佔位行**（圖由用戶後續手動補）。
- **D19**：用戶於 prompt 中明示時方寫圖引用。
- **D20**：文字內容須能獨立成立；不寫「詳如附圖」、「請參見圖示」等依賴圖之承載句型。

### 3.9 參考文獻

- **D21**：服務建議書屬提案文件，非學術論文。**無真實可查證來源時一律寫「無」**。
- **D22**：保留之參考文獻須具備下列其中一項具體性：
  - 具體報告編號／文件編號（如 `TPC-HLW-XXX`、`NUMO-TR-XX-XX`）
  - 具體發布年度與版次
  - 具體法規條號與公布日期
  - 具體標準號與年份（如 `ISO 19115:2014`）
  - 具體期刊／專書之卷期頁碼
- **D23**：僅有機構名稱與通稱（如「SKB Technical Reports」、「IAEA Specific Safety Guides」）而無具體性者，**不得列入**；湊字數之偽參考文獻反而減損專業度。

### 3.10 標點全形化

- **D24**：標點全形（`，。；：「」（）`）；中英文之間不加空格；括號半形 `()`。
- **D25**：標點檢核可用以下節點檢視：
  - 全形逗號 vs 半形：所有 CJK 字之間之逗號應為全形「，」
  - 全形分號 vs 半形：所有 CJK 字之間之分號應為全形「；」
  - 全形冒號 vs 半形：所有 CJK 字之間之冒號應為全形「：」
  - 例外：英文括號內（如 `Data Management Plan, DMP`）之逗號為半形

## 四、Don't 清單（角色邊界禁用詞與禁用句型）

### 4.1 治理流程動作類禁用（呼應 L2 §三）

| 禁用 | 緣由 |
|---|---|
| 建構審查流程 | 治理屬業主責任，乙方主動建構＝越界承諾 |
| 訂定發布機制 | 同上 |
| 設計治理框架 | 同上 |
| 建立審認同意機制 | 同上 |
| 主導發布狀態管理 | 同上 |
| 規劃跨單位治理流程 | 同上 |

### 4.2 後台帶治理意涵之功能類禁用

| 禁用 | 緣由 |
|---|---|
| 送審 | 暗示乙方建構審查流程 |
| 下架 | 暗示乙方建構發布管理流程 |
| 發布狀態管理 | 暗示治理化 |
| 審認同意機制 | 暗示乙方建構治理 |
| 發布審查流程 | 同上 |

**後台允許之純技術操作**（不含治理意涵）：

- 草稿、預覽、儲存、刪除、編譯導出
- 上傳、編輯、新編修刪
- 視覺化編輯器、頁面編排組件

### 4.3 大餅承諾類禁用

| 禁用 | 緣由 |
|---|---|
| 年度技術盤點與升級評估之常設節點 | 承諾系統性機制超出當期需求 |
| 跨單位治理框架 | 攬下不屬乙方之責任 |
| 年度法規盤點機制 | 系統性承諾，中選後被要求兌現 |
| 五年期分年調整全套機制 | 同上 |
| 跨節治理主軸 | 此詞本身亦不得出現於正文 |

### 4.4 絕對日期作自我承諾禁用

詳見 L2 §4.1。

### 4.5 寫死人員、子廠商、預算之禁用

詳見 L2 §4.2。

### 4.6 比喻成語類禁用

呼應 L1 §3.1：不另起爐灶／另起爐灶／重起爐灶／打掉重練／砍掉重練／白手起家／另立山頭。

### 4.7 流行管理術語類禁用

呼應 L1 §3.2：滾動／滾動式／滾動調整／滾動修正／滾動推進／滾動更新／滾動補強。

### 4.8 非中文符號禁用

呼應 L1 §3.3：→、⇒、⟶。

### 4.9 推卸式/補充式引導句禁用

呼應 L1 §3.4 與 §3.5：補充如後／另補充如後／茲補充說明／另作補充；差異點在於／不同之處在於／差異在於（於業主已議定子項描述前）；不另贅述／詳見他處／以下分述之；詳見他章。

### 4.10 第一、第二人稱禁用

我、我們、你、您。

## 五、灰色詞使用規則

下列詞語介於「純技術設計」與「治理流程」之間，**看寫法定可否**：

| 詞語 | 可寫之寫法（純技術／被動配合）| 不可寫之寫法（攬治理） |
|---|---|---|
| 版次紀錄／變更紀錄 | 作為資料庫之版本管理功能 | 「乙方建構版次審查流程」 |
| 身分驗證／權限控管 | 與既有單一登入系統介接 | 「乙方建構身分治理框架」 |
| 年度檢討／適時調整 | 「於需求變更時提請會議檢討」 | 「乙方建立年度盤點機制」 |
| 資料分流／內外分流 | 「內部與對外資料於系統層分流」（純架構設計）| 「乙方建構分流審查機制」 |
| 審查狀態／審查意見 | 「依業主審查意見修訂」 | 「乙方建立審查狀態管理流程」 |

### 5.1 判斷準則

- 此寫法是否描述「乙方做什麼技術動作」？若是，可寫。
- 此寫法是否描述「乙方建構治理機制」？若是，違反角色邊界，須改寫。

## 六、本案專有名稱規則範本

> 以下為**通用範本**，標註 `【】` 之處於各案啟動時依當期招標需求與既有契約另行填入；專案實際使用時建議另存為 case 專屬檔案，不於本技能檔內寫入特定案件名稱。

### 6.1 命名規則

- **【內部系統全稱】**：全文強制全稱；內文不可任意簡稱（如「資料庫系統」等模糊形式）。理由：策略性鎖定承接範圍，避免中選後被以模糊名義將其他系統爛攤子塞進來。
- **【公開網站全稱】**：首次以全稱書寫；同一段落內後續引用**可簡稱**為【公開網站簡稱】以利讀感流暢（簡稱由該案 case 檔指定）。
- **【工作名全稱】**：強制全稱。
- **業主／招標單位／發包與窗口**：均寫【業主全稱】，不得簡稱（不可寫【業主簡稱】、「本公司」、「甲方」等）。
- **主計畫名稱不得帶出**：本稿不得提及主計畫之全稱（包括【主計畫全稱】）。

### 6.2 章節標題之命名

- 章節標題遇招標原文用無前綴名稱（如「資料庫系統功能擴充與管理」）時，**沿用招標原文**（不主動加領域前綴）；內文方於需要時補上領域前綴之全稱。
- 此一致性確保招標清單對齊原則之「不得改名」於章節標題層面也成立。

### 6.3 內部系統 vs 公開網站之簡稱寬嚴差異

- **內部系統**：強制全稱，全文無例外，**不得在段落內簡稱**。理由：此為本工作核心擴充對象，全稱本身即承擔識別功能；策略性鎖範圍。
- **公開網站**：首次以全稱書寫；**同一段落內後續引用可簡稱**，以利讀感流暢。理由：公開網站之全稱往往過長，本地簡稱不影響識別。

### 7.1 提示詞模板（極簡版）

```markdown
請使用技能 role-writer-report 撰寫服務建議書（即套用 L1 角色設定 + L2 報告設定-服務建議書 + L3 撰寫設定-服務建議書）。

- ...（依當期案件填入，對應 L3 §六之 placeholder）
```

### 7.2 多 agent 協作

可同時派出 Claude + Codex 等多個 CLI agent 各自獨立產製，再融合：

1. 寫共用 prompt 檔案
2. 平行 dispatch（背景執行）
3. 等候完成
4. 融合各 agent 之優秀段落
5. 標點與禁用詞校核
