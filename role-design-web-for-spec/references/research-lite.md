# 輕量研究前置（Research Lite）

此檔定義 role-design-web-for-spec 在動手前要做的**最少量**用戶理解。

**核心原則**：不做學術級研究，不招募參與者，不跑統計。只做「足以讓設計決策有依據、而非憑空」的輕量描述，並把假設攤開讓使用者糾正。

---

## 何時做、何時跳過

| 情境 | 做輕量研究？ |
|---|---|
| 使用者已提供 Persona / Journey / 訪談紀錄 | ❌ 直接採用 |
| 使用者有完整 PRD + 明確目標用戶描述 | ❌ 直接上 |
| 使用者有 PRD 但沒提用戶 | ✅ 寫 5 欄位 Persona |
| 使用者只給一句話（「做個登陸頁」） | ✅ 至少釐清主要用戶＋主要任務＋情緒目標 |
| codebase 已有產品要擴充 | ⚠️ 讀現有頁面推論 Persona 假設，向使用者驗證 |
| 純 UI 探索（「試試這個配色方向」） | ❌ 跳過 |
| 純風格模仿（「做得像 Linear」） | ❌ 跳過 |

---

## 輕量 Persona 模板（5 欄位）

刻意做簡，避免變成 jnMetaCode UI Designer 那種長篇規格。

```markdown
## Persona：[一句話稱呼，如「忙碌的財務長」]

- **角色**：[職業 / 身份 / 與產品關係]
- **場景**：[典型使用情境：何時、何處、用什麼裝置]
- **主要任務**：[他們來這產品要做的第一件事]
- **痛點**：[目前方法的最大挫折]
- **情緒目標**：[離開產品時想帶走的感受，如「安心」「快速完成」「被啟發」]
```

**規則**：
- **每個產品最多 2 個 Persona**（主要＋次要）。多了就失焦。
- **必須標註「這是我的假設」**，不要寫成既成事實。
- 寫完後**逐欄位問使用者**：「這個推論對嗎？」

## 輕量 Journey Map 模板

只畫**核心任務路徑**，不畫整個產品。

```markdown
## Journey：[任務名稱，如「第一次匯入資料」]

| 階段 | 動作 | 想法 | 情緒 | 痛點 / 機會 |
|---|---|---|---|---|
| 1. 發現 | ... | ... | 😐 | ... |
| 2. 進入 | ... | ... | 🙂 | ... |
| 3. 執行 | ... | ... | 😣 | ⚠️ [此處設計該解的問題] |
| 4. 完成 | ... | ... | 😊 | ... |
```

**規則**：
- **只畫會影響當前設計的段落**，不畫 end-to-end。
- 情緒用 emoji 做視覺錨定（這是內部文件的例外，不是最終產物）。
- 痛點 / 機會欄位**要能連到具體的 UI 決策**（如「第 3 段痛點 → 主介面需要明顯的進度指示」）。

## 快速 Task Hypothesis 模板

當連 Persona 都太重時，用這個最輕量版本：

```markdown
## Task Hypothesis

- **使用者是誰**：[一句話]
- **他們想做什麼**：[主要任務，一句]
- **完成的樣子**：[成功完成時的狀態，一句]
- **會放棄的樣子**：[何時會關掉頁面離開，一句]
```

**規則**：四句話。寫超過表示想太多了，問使用者。

## 研究假設如何接回設計決策

Step 0 寫下的假設 → Step 3 的 Design System 宣告要引用 → Step 5 完整實作時要對照。

**範例**：

```markdown
## Step 0 假設
- Persona：忙碌的財務長
- 情緒目標：5 分鐘內找到想看的數字，安心離開

## Step 3 Design System 回應
- 配色：冷靜中性為主（藍灰），避免花俏 → 對應「安心」
- 字級：數字要大（h1 64px） → 對應「快速找到」
- 動效：極簡過場（≤ 200ms） → 對應「不干擾」
- 變體：密集版（財務長）vs 總覽版（CEO） → 對應兩個 Persona 差異

## Step 5 實作驗證
- 開測試頁面時，5 秒內找到核心數字 → ✅ 與假設一致
- 動效沒有干擾閱讀 → ✅
```

**沒有 Step 0 → Step 3 → Step 5 的連接，就是憑空設計。**

## 研究知識累積（可選）

跨 session / 跨專案累積輪廓。建議寫到使用者的長期記憶或專案 memory：

- 此使用者的產品類型偏向哪些 Persona（B2B SaaS、消費性 App、內部工具）
- 他們常提到的用戶情緒目標（效率、信任、驚艷、教育）
- 他們通常跳過研究的情境 vs 會認真對待的情境

積累**使用者偏好**比積累「普世 Persona」更有價值 —— 你是為這個使用者服務，不是寫學術論文。

---

## 什麼是「過度研究」

避免這些：

- ❌ 寫超過 5 個欄位的 Persona（變成 UX Researcher 規格）
- ❌ 幫使用者做訪談腳本（那不是設計工程師的職責）
- ❌ 畫 end-to-end Journey（超出當前設計範圍）
- ❌ 提供統計樣本量建議（超出輕量研究）
- ❌ 寫研究報告 Markdown 交付模板（那是 UX Researcher 的產物）

**role-design-web-for-spec 是設計工程師，不是 UX Researcher。** 研究前置的目的是**讓當前設計不憑空**，而不是**交付研究成果**。
